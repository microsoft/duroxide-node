# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [0.1.26] - 2026-05-13

### Fixed

- **Entra HTTPS token acquisition** тАФ picks up the upstream native-TLS fix in
  `duroxide-pg 0.1.33` ([microsoft/duroxide-pg#8]). Released `duroxide-pg 0.1.32`
  pulled `reqwest` transitively without a TLS connector compiled in, which
  caused every HTTPS call (including AAD token acquisition for
  `connectWithEntra` / `connectWithSchemaAndEntra`) to fail with
  `error sending request` / `invalid URL, scheme is not http`. No public API
  changes.

### Changed

- **Bumped `duroxide-pg` dependency** тАФ `0.1.32` тЖТ `0.1.33`.

[microsoft/duroxide-pg#8]: https://github.com/microsoft/duroxide-pg/pull/8

## [0.1.25] - 2026-05-09

### Added

- **Microsoft Entra ID authentication** тАФ `PostgresProvider.connectWithEntra(host, port, database, user, options?)` and `PostgresProvider.connectWithSchemaAndEntra(host, port, database, user, schema, options?)` connect to Azure Database for PostgreSQL Flexible Server via Entra access tokens. Credentials are resolved via the default chain (WorkloadIdentityCredential тЖТ ManagedIdentityCredential тЖТ DeveloperToolsCredential). A background task refreshes the token before expiry. All connections use `PgSslMode::VerifyFull`.
- **`PostgresEntraOptions` interface** тАФ optional tunables: `audience`, `maxConnections`, `acquireTimeoutMs`, `refreshIntervalMs`. All fields default to sensible Azure public-cloud values.

### Changed

- **Bumped `duroxide` dependency** тАФ `0.1.28` тЖТ `0.1.29`
- **Bumped `duroxide-pg` dependency** тАФ `0.1.30` тЖТ `0.1.32`



### Added

- **Linux ARM64 GNU package** - Publish `duroxide-linux-arm64-gnu`
  for `aarch64-unknown-linux-gnu` and include it as an optional dependency so
  `linux/arm64` Node.js containers load the native binding.
- **Linux ARM64 CI smoke** - Build, pack, publish, and smoke-test Linux ARM64
  on `ubuntu-24.04-arm`.

## [0.1.23] - 2026-04-23

### Fixed

- **Windows packaging** тАФ `index.js` was regenerated by `napi build` during
  the 0.1.22 release and lost the manual `require('duroxide-windows-x64')`
  patch from #13, falling back to the (non-existent) napi-default name
  `duroxide-win32-x64-msvc`. Re-apply the fix; 0.1.22 was never published
  because the cross-platform smoke test (windows-latest) caught it.

## [0.1.22] - 2026-04-23

- **Bumped `duroxide-pg` dependency** тАФ `0.1.29` тЖТ `0.1.30` (same `ring`-drop cascade)

> тЪая╕П **Do not use 0.1.22** тАФ npm publish was skipped due to a Windows

## [0.1.21] - 2026-04-17

### Fixed
- `npm install duroxide` now correctly pulls the `duroxide-windows-x64` optional
  dependency. In `0.1.20` the published `optionalDependencies` in the main
  package were rewritten by napi-rs's `prepublish` step back to the default
  `duroxide-win32-x64-msvc` name, which does not exist on npm post-rename, so
  Windows installs failed with `MODULE_NOT_FOUND`. Caught by the new
  post-publish registry smoke. CI now publishes the main package with
  `--ignore-scripts` to preserve our manifest exactly.

## [0.1.20] - 2026-04-17

### Fixed
- Windows dispatcher now resolves the correct platform subpackage (`duroxide-windows-x64`).
  Previously `require('duroxide-win32-x64-msvc')` in `index.js` threw `MODULE_NOT_FOUND`,
  making `duroxide@0.1.19` unusable on Windows ([microsoft/duroxide#13](https://github.com/microsoft/duroxide/issues/13)).

### Added
- Cross-platform packaging smoke tests in `ci/smoke/` that install from local tarballs on
  ubuntu, macos-14, macos-13, and windows-latest, plus post-publish registry smoke. Publish
  is gated on all-platform smoke success тАФ partial platform releases are no longer possible.

## [0.1.7] - 2026-02-20

### Added
- **Custom Status API**: `ctx.setCustomStatus(status)` and `ctx.resetCustomStatus()` тАФ set/clear custom status from orchestrations (fire-and-forget, no yield needed)
- **Custom Status on OrchestrationStatus**: `customStatus` and `customStatusVersion` fields on status objects returned by `getStatus()`, `waitForOrchestration()`, etc.
- **Custom Status Polling**: `client.waitForStatusChange(instanceId, lastSeenVersion, pollIntervalMs, timeoutMs)` тАФ poll for custom status changes
- **Event Queue API (Context)**: `ctx.dequeueEvent(queueName)` тАФ dequeue from a named FIFO mailbox (survives continue-as-new)
- **Event Queue API (Client)**: `client.enqueueEvent(instanceId, queueName, data)` тАФ enqueue to an instance's event queue
- **Retry with Session**: `ctx.scheduleActivityWithRetryOnSession(name, input, retry, sessionId)` тАФ retry with session affinity
- `scheduleActivityWithRetry` now respects `sessionId` when present
- EventKind support for `QueueSubscribed`, `QueueEventDelivered`, `QueueSubscriptionCancelled`

### Changed
- Upgraded duroxide to 0.1.19, duroxide-pg to 0.1.21

### Deprecated
- `client.raiseEvent()` for persistent messaging тАФ use `client.enqueueEvent()` instead
- `ctx.waitForEvent()` for persistent subscriptions тАФ use `ctx.dequeueEvent()` instead

## [0.1.0] - 2026-02-10

### Added

- **Core SDK** тАФ Node.js wrapper around the Rust duroxide runtime via napi-rs
  - Generator-based orchestrations (`function*` + `yield`) for deterministic replay
  - Async activities (`async function` + `await`) for side effects
  - Structured tracing delegated to Rust (auto-suppressed during replay)

- **Orchestration Context API**
  - `scheduleActivity()`, `scheduleActivityWithRetry()` тАФ activity scheduling with optional retry policies
  - `scheduleTimer()` тАФ durable delays that survive restarts
  - `waitForEvent()` тАФ pause for external signals
  - `scheduleSubOrchestration()`, `scheduleSubOrchestrationWithId()` тАФ child workflows
  - `startOrchestration()` тАФ fire-and-forget orchestration launches
  - `all()` тАФ parallel fan-out/fan-in supporting all task types (activities, timers, waits, sub-orchestrations)
  - `race()` тАФ first-to-complete between 2 tasks, supporting all task types
  - `utcNow()`, `newGuid()` тАФ deterministic values for replay safety
  - `continueAsNew()` тАФ restart orchestrations with fresh history
  - Nested `all()`/`race()` rejection with clear error messages

- **Activity Context API**
  - `isCancelled()` тАФ cooperative cancellation (detects when orchestration no longer needs the result)
  - `traceInfo()`, `traceWarn()`, `traceError()`, `traceDebug()` тАФ structured tracing with full metadata
  - Context fields: `instanceId`, `executionId`, `orchestrationName`, `activityName`, `workerId`

- **Runtime Options**
  - `orchestrationConcurrency`, `workerConcurrency`, `dispatcherPollInterval`
  - `workerLockTimeoutMs` тАФ controls activity cancellation detection latency
  - `observability` тАФ log level, service name, gauge polling interval

- **Client API**
  - `startOrchestration()`, `startOrchestrationVersioned()` тАФ launch workflows
  - `waitForOrchestration()` тАФ poll until completion with timeout
  - `raiseEvent()` тАФ send external events to waiting orchestrations
  - `cancelInstance()` тАФ cancel running orchestrations
  - `getOrchestrationStatus()` тАФ query status without waiting
  - `deleteInstance()` тАФ remove completed instances
  - `getSystemMetrics()`, `getQueueDepths()` тАФ observability
  - `listAllInstances()` тАФ admin listing

- **Storage Providers**
  - SQLite (`SqliteProvider`) тАФ file-backed or in-memory, good for development/testing
  - PostgreSQL (`PostgresProvider`) тАФ production multi-node deployments with schema isolation

- **Versioning** тАФ `registerOrchestrationVersioned()` for safe rolling upgrades

- **50 tests** across 4 test suites
  - `e2e.test.js` тАФ 23 PostgreSQL e2e tests + 1 SQLite smoketest
  - `races.test.js` тАФ 7 tests for `ctx.all()` and `ctx.race()` with mixed task types + cooperative cancellation
  - `admin_api.test.js` тАФ 14 admin/management API tests
  - `scenarios/toygres-node.test.js` тАФ 6 scenario tests modeling real-world patterns
